21 /*
22  *
23  * This was originally part of the group documentation, but it was removed,
24  * since it's not important for connect platforms. It is kept as it might become
25  * important later when we add support of new platforms.
26  *
27  * The base time units for events are ticks. Each tick is approximately equal
28  * to a millisecond, but the true duration depends on the platform. The
29  * duration of a tick is 1000 / ::MILLISECOND_TICKS_PER_SECOND, where 1000 is
30  * the number of milliseconds per second and ::MILLISECOND_TICKS_PER_SECOND is
31  * the platform-specific number of ticks per second. For example,
32  * ::MILLISECOND_TICKS_PER_SECOND on the EM357 SoC is 1024, so each tick is
33  * 1000 / 1024 = ~0.98 milliseconds. Calling
34  * ::emberEventControlSetDelayMS(someEvent, 100) on the EM357 SoC will schedule
35  * the event for 100 ticks * (1000 milliseconds / 1024 ticks) =
36  * ~97.7 milliseconds. Note, however, that the accuracy of the base tick
37  * depends on the timer source. Furthermore, the scheduled delay is the minimum
38  * delay. If ::emberRunEvents or ::emberRunTask are not called frequently
39  * enough, the actual delay may be longer than the scheduled delay.
40  *
41  * Additionally, the APIs for quarter second and minute delays
42  * (::emberEventControlSetDelayQS and ::emberEventControlSetDelayMinutes) use
43  * "binary" units. One quarter second is 256 ticks and one minute is 65536
44  * ticks. Calling ::emberEventControlSetDelayMinutes(someEvent, 3) on the
45  * EM357 SoC will schedule the event for 3 minutes * (65536 ticks / minute) *
46  * (1000 milliseconds / 1024 ticks) = ~3.2 minutes. It is possible to avoid
47  * these binary units by using ::emberEventControlSetDelayMS and the various
48  * MILLISECOND_TICKS_PER_XXX multipliers. For example, calling
49  * ::emberEventControlSetDelayMS(someEvent, 3 * MILLISECOND_TICKS_PER_MINUTE)
50  * will delay for 3 minutes on any platform. Be aware of
51  * ::EMBER_MAX_EVENT_CONTROL_DELAY_MS when using this approach.
52  */
